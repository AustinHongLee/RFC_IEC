## 1. 架構原則(全程適用)

- **加法式遷移,絕不破壞既有資料。** 一律 `ALTER TABLE ADD COLUMN` 或新建表;不 drop、不重命名既有欄位(`DB數` 等公司慣用名保留)。
- **白名單同步。** 每加一個可寫欄位 → 更新 `app.py` 的 `*_COLS`。
- **匯入器冪等。** 重匯同一專案代號會覆蓋該案(現行邏輯 `replace=True`),新欄位要能重匯帶入,也要能對已匯入資料補值。
- **單檔 SQLite 不變。** 維持「複製 `weld.db` 即備份」。多人同時才升 PostgreSQL(階段 7),屆時表結構沿用。
- **UI 沿用既有產生器。** 新表單走 `jointFields()` 風格的欄位陣列 + `formHTML/collectForm`;新分頁走 `nav.tabs` + `renderXxx()`。
- **每階段可獨立上線。** 完成一階段就能用,不必等後面。

## 階段 0 — 遷移機制 + 種子(前置,0.5–1 人日)

**為什麼必做:** `init_db()` 對既有表不會加欄位。後面每一個要 `ADD COLUMN` 的階段都靠它。沒有它,改了 `schema.sql` 但 `weld.db` 不會變。

**做法(`db.py`,沿用無 ORM 風格):** 用 `PRAGMA user_version` 當版本號,維護一個有序遷移清單,`init_db()` 末段套用。

```python
# db.py — 在 init_db() 之後新增
def _column_exists(conn, table, col):
    return any(r[1] == col for r in conn.execute(f"PRAGMA table_info({table})"))

def _add_column(conn, table, col, decl):
    if not _column_exists(conn, table, col):
        conn.execute(f"ALTER TABLE {table} ADD COLUMN {col} {decl}")

# 有序遷移:每個是 (版本, 函式)。只增不改。
MIGRATIONS = [
    # (1, lambda c: _add_column(c, "weld_joint", "spool_id", "INTEGER REFERENCES spool(id)")),
    # 後續階段往這裡加
]

def run_migrations():
    with get_conn() as conn:
        ver = conn.execute("PRAGMA user_version").fetchone()[0]
        for v, fn in MIGRATIONS:
            if v > ver:
                fn(conn)
                conn.execute(f"PRAGMA user_version = {v}")
```

`init_db()` 結尾呼叫 `run_migrations()`(先 `executescript(schema.sql)` 建新表,再跑 ALTER)。新表照常寫進 `schema.sql`(`CREATE TABLE IF NOT EXISTS`),欄位新增寫進 `MIGRATIONS`。

**驗收:** 對現有 `weld.db`(DEMO/5379)跑一次,`PRAGMA user_version` 前進、既有資料筆數不變、新欄位出現且為 NULL。
**風險:** 低。SQLite `ADD COLUMN` 不支援 `IF NOT EXISTS`,故用 `_column_exists` 守門,可重複執行不報錯。

## 階段 2 — DB數 自動計算(最快見效,1–2 人日)

**目標(藍圖 §2):** 新增/匯入焊口時,`db_count` 留空則自動帶 `max(1, 尺寸吋) × 係數`。係數取 `weld_joint.db_factor`(預設 1)。

**核心:尺寸→吋 解析器**(新檔 `sizes.py`,匯入器與 API 共用)。要吃得下 `1.5"`、`1-1/2"`、`½"`、`¾"`、`3/4"`、`DN50`(可選)等寫法。

```python
# sizes.py
import re
_FRAC = {'½':0.5,'¼':0.25,'¾':0.75,'⅜':0.375,'⅝':0.625,'⅞':0.875,'⅛':0.125}
def parse_inch(s):
    if s is None: return None
    s = str(s).strip().replace('＂','"').replace('”','"')
    for u in _FRAC:  # unicode 分數
        s = s.replace(u, f'+{_FRAC[u]}')
    s = s.replace('"','').replace('inch','').replace('IN','').strip()
    s = s.replace('-', '+') if re.match(r'^\d+\+', s) or re.match(r'^\d+-\d+/\d+', str(s)) else s
    m = re.findall(r'\d+/\d+|\d+\.?\d*', s)
    if not m: return None
    total = 0.0
    for tok in m:
        total += eval(tok) if '/' in tok else float(tok)  # tok 受限於數字/斜線
    return total or None

def db_count(size, factor=1):
    inch = parse_inch(size)
    if inch is None: return None
    return max(1.0, inch) * (factor or 1)
```

**接入點**

- `app.py::create_joint` / `update_joint`:`data` 若無 `db_count`(或為 None)且有 `size` → `data["db_count"] = sizes.db_count(data.get("size"), data.get("db_factor",1))`。
- `importer.py`:`_f(cell(... "db_count"))` 為 None 時,以 `sizes.db_count(size, factor)` 回填(目前 cp-129 有現成 DB數/係數,se-39、gl-05 部分缺係數→factor 預設 1)。
- 新增回填 API:`POST /api/projects/{pid}/joints/recompute-db`(payload `{only_blank: true}`),把該案 `db_count IS NULL` 者批次補上;回傳補了幾筆。前端「焊口登錄」加一顆「補算 DB數」按鈕。

**UI:** `jointFields()` 的 `db_count` 欄 placeholder 改為「留空自動計算」。

**驗收(已可用真實資料驗):** 對 cp-129 全 5,379 口,用解析器重算與來源 `DB數` 欄比對,應 100% 相符(藍圖已宣稱)。本計畫附帶的驗證腳本已先跑過(見文末驗證註記)。
**風險:** 中低。風險集中在尺寸字串怪寫法;以「比對既有 DB數」當回歸測試把關。`eval` 僅作用於 `\d+/\d+` 已過濾字串,安全。

## 階段 6 — 資料補強(1.5–2.5 人日)

**目標(藍圖 §6、§8):** 修正 se-39 完成率顯示 0%;把預製掃描日期接入圖面。

**根因(已查證):** 匯入器只讀 `焊口編號明細`。se-39 的該分頁(`焊口編號明細--2025.11.28-v2`)沒有完成日期欄;完成日期在另兩個分頁。

**做法**

1. **匯入器加「完成資料合併」步驟。** 在 `import_weld_control` 寫完焊口後,若工作簿含 `完成焊口建檔-v2`(或 `已申報完成焊口明細`),讀其 `流水號 + 銲口編號 → 配管完成日期 / S/F / 施工廠商`,以 (drawing serial/no, joint_no) 為鍵 `UPDATE weld_joint SET weld_date=?, shop_field=COALESCE(shop_field,?), subcontractor=?, status='完成' WHERE ...`。新增別名群組 `COMPLETION_ALIASES`,新增 `pick_sheet(wb, "完成焊口建檔-v2", lambda n:n.startswith("完成焊口建檔"), "已申報完成焊口明細")`。
2. **掃描日期。** 讀 `預製掃描圖` 分頁(欄:分類、file、sn、完成焊口數、LINE NO.),以 `sn`/流水號對到 `drawing.serial_no`,回填 `drawing.scan_date`(若該分頁無日期欄,改以「有出現在清單=已掃描」設旗標,或抓檔名日期)。cp-129 的 `ISO圖號清單` 已有 `預製圖掃描日期` 欄,現行 `DRAWING_ALIASES["scan_date"]` 已涵蓋。
3. **一次性修補現有 se-39。** 提供 `tools/merge_se39_completion.py`:對既載入的 se-39 專案,直接讀來源檔的完成分頁批次 UPDATE(不必整案重匯)。同樣邏輯也可做成 API `POST /api/projects/{pid}/merge-completion`(上傳同一 Excel)。
4. 合併後重跑「請款 自動歸期」即可得各期 DB數(來源檔 `請款期別` 分頁:配管完成日期→請款期別,與現行 `billing_auto_assign` 邏輯一致)。

**SE-37 / gl-05 未分類系統(§8):** 來源無乾淨「系統」欄。提供由 `line_no` 前綴推導系統的小工具:`UPDATE pipe_line SET system_id=...`,或在「基礎資料」手補。屬選配。

**驗收:** se-39 合併後,儀表板「完成率(口數/DB數)」> 0 且與來源檔 `PV.系統overview` 的完成 DB 數量級相符;`drawing.scan_date` 非空比例上升。
**風險:** 中。鍵對應(流水號 vs 圖號、民國日期、重複焊口號)要小心;`完成焊口建檔-v2` 有 `重複輸入檢查` 欄可參考。先在複本 db 跑、比對筆數再正式套。

## 階段 1 — Spool 層(核心,3–5 人日)

**目標(藍圖 §1、§6):** 圖面 → **spool** → 焊口;焊口可歸屬 spool,spool 滾動彙總 S/F、進度、狀態,支撐「最大化廠內預製」策略。

**Schema(新表寫 `schema.sql`;新欄位走階段 0 遷移)**

```sql
CREATE TABLE IF NOT EXISTS spool (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id  INTEGER NOT NULL REFERENCES project(id) ON DELETE CASCADE,
  drawing_id  INTEGER REFERENCES drawing(id) ON DELETE CASCADE,
  spool_no    TEXT NOT NULL,              -- 廠內 spool 編號
  shop_field  TEXT DEFAULT 'S',          -- 此 spool 性質 預製S/現場F
  status      TEXT DEFAULT '規劃',        -- 規劃/下料/組對/銲接/NDE/油漆/完成/出貨
  fab_dwg_no  TEXT,                       -- 預製圖號
  scan_date   TEXT,                       -- 預製圖掃描日期
  ship_date   TEXT,                       -- 出貨到場日
  remark      TEXT,
  created_at  TEXT DEFAULT (datetime('now','localtime')),
  UNIQUE(project_id, drawing_id, spool_no)
);
CREATE INDEX IF NOT EXISTS idx_spool_drawing ON spool(drawing_id);
```

遷移(階段 0 `MIGRATIONS`):`_add_column(c, "weld_joint", "spool_id", "INTEGER REFERENCES spool(id)")`;`CREATE INDEX idx_joint_spool ON weld_joint(spool_id)`。

**後端**

- 白名單:`JOINT_COLS` 加 `spool_id`;新增 `SPOOL_COLS = {"drawing_id","spool_no","shop_field","status","fab_dwg_no","scan_date","ship_date","remark"}`。
- API:`GET/POST /api/projects/{pid}/spools`(列表帶每 spool 焊口數/完成數/DB數)、`PUT/DELETE /api/spools/{sid}`、`GET /api/drawings/{did}/spools`、`POST /api/spools/{sid}/assign`(payload `{joint_ids:[...]}` 批次設 `spool_id`)。
- 儀表板擴充:`by_spool_status`、預製 vs 現場 口數/DB數(用既有 `shop_field` 即可算,spool 讓它更細)。
- `v_joint_full` view 加 `LEFT JOIN spool` 帶出 `spool_no`(view 重建免遷移)。

**前端**

- 新分頁 `spools`(`index.html` `nav.tabs` 加按鈕 + `<section id="view-spools">`;`app.js` 加 `renderSpools()` 並掛進 `renderView` 對應表)。
- 圖面列可展開顯示其 spool;焊口表單 `jointFields()` 加 `spool_id` 下拉(依目前 `drawing_id` 動態載入該圖 spool)。
- 焊口列表加「S/F」「spool」欄與篩選(沿用現有 filter 列)。

**資料遷移/回填:** 既有焊口 `spool_id` 留 NULL(未歸 spool,合法)。提供選配 helper:對每張圖的「預製 S」焊口自動建一個預設 spool 並掛上(`spool_no = <serial>-S01`),現場 F 焊口留無 spool。預設不自動跑,給按鈕。

**風險:** 中(改動面最大:schema+API+UI+view)。緩解:spool_id 可空、純加法;先在 DEMO 案手動建幾個 spool 做端對端測試。

## 階段 5 — 試壓包 + 品保卷冊 MDR / Turnover(3–5 人日)

**目標(藍圖 §5、§6):** 試壓包正式化、punch list、品保移交卷冊(MDR)。

**現況:** `weld_joint.test_package/test_date/test_result` 為平面欄(夠記單口結果),缺「包」這個聚合物件與 punch、MDR。

**Schema(新表)+ 遷移**

```sql
CREATE TABLE IF NOT EXISTS test_package (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id INTEGER NOT NULL REFERENCES project(id) ON DELETE CASCADE,
  pkg_no TEXT NOT NULL, kind TEXT,           -- 水壓/氣壓
  pressure TEXT, medium TEXT, test_date TEXT,
  result TEXT, reinstated INTEGER DEFAULT 0, remark TEXT,
  UNIQUE(project_id, pkg_no)
);
CREATE TABLE IF NOT EXISTS punch_item (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id INTEGER NOT NULL REFERENCES project(id) ON DELETE CASCADE,
  test_package_id INTEGER REFERENCES test_package(id) ON DELETE CASCADE,
  category TEXT, description TEXT, status TEXT DEFAULT '待處理', remark TEXT
);
CREATE TABLE IF NOT EXISTS mdr_document (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id INTEGER NOT NULL REFERENCES project(id) ON DELETE CASCADE,
  doc_type TEXT, title TEXT, ref_no TEXT, status TEXT DEFAULT '待彙整',
  file_path TEXT, remark TEXT
);
```

遷移:`_add_column(c,"weld_joint","test_package_id","INTEGER REFERENCES test_package(id)")`(取代字串 `test_package` 的關聯;舊欄保留為文字備份)。

**後端/前端:** 試壓包 CRUD + 指派焊口(`{joint_ids}` 設 `test_package_id`);新分頁「試壓 / 卷冊」顯示包、punch、MDR;儀表板加試壓完成率、未結 punch 數。

## 階段 4 — 採購模組 MTO→PR→PO→收料→領料(最大,6–10 人日)

**目標(藍圖 §4):** 物料控制鏈 + 料況看板 + 缺料預警。**相依階段 3 的 `material_heat`(收料對爐號)。**

**Schema(新表,分段交付)**

```sql
CREATE TABLE IF NOT EXISTS mto_item (        -- 物料需求(由圖面/MTO)
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id INTEGER NOT NULL REFERENCES project(id) ON DELETE CASCADE,
  drawing_id INTEGER REFERENCES drawing(id), item_name TEXT, spec TEXT,
  material TEXT, size TEXT, schedule TEXT, qty REAL, unit TEXT,
  source TEXT,                                -- 自購/業主供料
  status TEXT DEFAULT '需求'                  -- 需求/已請購/已採購/到料/缺料
);
CREATE TABLE IF NOT EXISTS purchase_req (id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id INTEGER NOT NULL REFERENCES project(id) ON DELETE CASCADE,
  pr_no TEXT, date TEXT, status TEXT, remark TEXT);
CREATE TABLE IF NOT EXISTS purchase_order (id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id INTEGER NOT NULL REFERENCES project(id) ON DELETE CASCADE,
  po_no TEXT, vendor TEXT, date TEXT, status TEXT, remark TEXT);
CREATE TABLE IF NOT EXISTS goods_receipt (id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id INTEGER NOT NULL REFERENCES project(id) ON DELETE CASCADE,
  grn_no TEXT, po_id INTEGER REFERENCES purchase_order(id),
  heat_id INTEGER REFERENCES material_heat(id), qty REAL, date TEXT, remark TEXT);
CREATE TABLE IF NOT EXISTS material_issue (id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id INTEGER NOT NULL REFERENCES project(id) ON DELETE CASCADE,
  mto_item_id INTEGER REFERENCES mto_item(id), qty REAL, date TEXT,
  issued_to TEXT, remark TEXT);
```

> gl-05 來源已有 `MTO表 / 採購表`(見 `GL-05-MTO_TABLE總表`),可作為 `mto_item` 匯入來源 — 另寫 `import_mto()`。

**交付切片:** 4a 物料需求(mto_item)+ 料況看板/缺料預警 → 4b PR/PO → 4c 收料(連爐號)→ 4d 領料。每片可獨立上線。
**驗收:** 一條 MTO 走完 需求→請購→採購→到料(掛爐號)→領料,料況看板顯示缺料。
**風險:** 高(範圍大、跨表)。緩解:分 4a–4d 切片;先做 4a 看板即有價值。
