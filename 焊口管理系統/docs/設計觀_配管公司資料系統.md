# 設計觀 — 一個配管公司的圖號/焊口/請款,怎麼從 Excel 變成好用的 SQLite 系統

> 回答三個寬問題:**該有哪些表、用什麼風格、要顧慮什麼。**
> 寫於 2026-06。可丟上 `docs/討論桌/` 讓各家 AI 補意見。

---

## 0. 一句話核心觀念

把 Excel 變成系統,**不是丟掉 Excel 的好用,而是留住 Excel 的「手感」(格子、篩選、排序、貼上、公式),底下換成一個有結構、不會亂、能跨表串起來的單一資料庫**。

> **資料只有一份真相 = SQLite;畫面盡量像 Excel 那樣好操作;Excel 退化成「匯入 / 匯出」的格式,不再是真相本身。**

這一句決定了後面所有取捨。

---

## 1. 一個配管公司「該有的表」(管制表 / 總表 / 請款表)

以「管線預製 + 現場安裝」公司來看,完整的一套大概是這 13 張。標注目前系統狀態:✅ 已有 / ◑ 部分 / ⬜ 還沒。

### 文件 / 圖面控制
1. **ISO 圖號管制表** ✅ — 圖號、內部流水號、版次、收圖日、transmittal、line no、系統、區域、P&ID、尺寸、第幾張/共幾張、圖種、狀態、預製圖掃描。
   *這是「文件控制」的根:每張圖的身分證。*

### 生產核心
2. **焊口管制表 / weld map** ✅ — 系統的心臟。每個焊口 + 全生命週期(規格、S/F、製程、焊工、NDE、PWHT、試壓、完成日、計量)。
3. **Spool 預製分段表** ✅ — 圖面 → spool → 焊口。廠內預製的最小單位;分 shop/field 是「最大化預製」策略的依據。

### 材料鏈
4. **MTO 物料清單** ⬜ — 每張圖的料、規格、數量(下料依據)。
5. **採購 / 料控** ⬜ — MTO → 請購 PR → 採購 PO → 收料 GRN → 倉儲 → 領料;料況看板、缺料預警。
6. **爐號 / MTR 追溯** ◑ — 母材爐號、材質證明、PMI、銲材批號;接到焊口兩側。(已有爐號/銲材登錄,尚缺採購端串接)

### 品質
7. **NDE / RT 清單** ◑ — 方法、比例、報告號、結果、底片張數、缺陷、補焊、重檢。
8. **PWHT 紀錄** ◑ — 曲線、溫度、時間、升降溫率。
9. **試壓包 + punch list** ⬜ — 試壓包編號、種類、壓力、結果、reinstatement、未結項。
10. **焊工 / WPS / PQR 資格** ◑ — 鋼印、證照到期、合格製程、連續性;WPS/PQR 庫。

### 商務
11. **請款 / 計價表** ✅ — 以**吋徑(DB數)**計進度與請款,分期別、單價、請款狀態。
12. **進度總表 / 跨案彙總** ◑ — 各系統/各案完成率、待辦、KPI。

### 移交
13. **品保卷冊 MDR / turnover** ⬜ — 把 weld map、MTR、NDE 報告、試壓紀錄、as-built 彙整成移交卷冊。

> 重點:**這些不是 13 個各自的 Excel,而是一個資料庫裡互相串起來的 13 個視圖**。同一個焊口,在「焊口管制表」「RT 清單」「請款表」「MDR」裡都會出現,但**資料只存一份**,各表只是不同的看法。這正是 Excel 做不到、而 SQLite 能做到的事。

## 2. 風格取捨 — 從 Excel 到 App,落在哪?

一條光譜,左邊最像 Excel、右邊最像傳統資料庫 App:

| 風格 | 好處 | 壞處 | 適合 |
|---|---|---|---|
| **純 Excel / 巨集** | 熟悉、彈性最大、零學習 | 沒結構、會亂、版本災難、量大崩、無法跨表串 | (你正在逃離的) |
| **試算表式格子(grid app)** | 保留格子手感 + 結構/篩選/關聯/權限;像 Airtable / Google Sheet 疊在資料庫上 | 要寫;太像 Excel 時使用者會想塞自由文字破壞結構 | **量大、扁平、批次**:完銲日期、焊工、NDE 結果跨幾千口 |
| **表單 / 紀錄式 App** | 結構最強、引導性好、適合複雜單筆 | 批次輸入慢;一次 40 欄會「讓人難受」(現況) | **單一焊口的複雜一生** |

**取捨軸:** 好上手 + 批次快(Excel/格子) ←→ 結構嚴 + 引導複雜資料(表單/App)。

### 建議落點:**混合**
- **清單 / 批次操作 → 用「格子(grid)」**:可內聯編輯、可多選一次填、可像 Excel 那樣篩選排序貼上。處理高頻、扁平、量大的更新。
- **單一焊口的生命週期 → 用「紀錄 / 時間軸」**:不要一次 40 欄;用「走到哪、誰做的、下一步填什麼」呈現。
- 這正是先前那張 UI mockup 的取向。

> 一句話:**把「Excel 最好用的部分(格子+篩選+批次)」留給量大的清單,把「App 最強的部分(結構+引導)」留給複雜的單筆。**

## 3. 需要考量的(做之前先想清楚)

1. **單一真實來源(SSOT)** — 進了資料庫,**DB 才是真相**,Excel 只進出。必須先解掉現在「重新匯入會覆蓋整個專案、洗掉手動輸入」的問題(改成:匯入只補新焊口/只更新指定欄、不蓋手填值)。**這是最關鍵、最該先決的一條。**
2. **識別碼要穩** — 圖號 / 流水號 / 焊口編號 是串表的鑰匙。格式不一致(數字 vs 文字、補零、半形全形)會讓跨表對不上(合併 se-39 完成資料時已踩過)。需要一套規範化規則。
3. **版次與增刪要留歷史** — 圖會改版、焊口會增減。不能直接覆蓋,要能回答「這口是哪一版加的、誰刪的」。
4. **計量公式內建** — DB數 = max(1, 尺寸吋) × 係數,由系統算、可稽核,不靠手填(已做)。
5. **規模 / 效能** — 單案 2.5 萬+ 焊口。篩選要建索引、清單要虛擬捲動、統計要能即時。
6. **還是要能匯出 Excel** — 業主送審、請款、對帳,很多時候仍吃 Excel。系統要能「把任何篩選結果一鍵匯出成 Excel」。
7. **備份 / 資料安全** — 單檔 SQLite 好備份(複製一個檔),但要有紀律(關 app 再備份;或 DB 不進 git、改用匯出備份)。剛踩過 weld.db 毀損。
8. **非工程師可用性** — 回到北極星:好輸入、直觀、可見、可精確篩選。技術偏好與這個衝突時,讓步。
9. **漸進遷移,不要大爆炸** — 別逼大家一夜之間從 Excel 全搬過來。讓系統先跑、與 Excel 平行一段時間,信任建立了再收掉 Excel。
10. **單機 vs 多人** — 目前單機 SQLite 夠用;十幾人同時即時編輯才需要升 PostgreSQL / 正式網頁。表結構已可沿用。
