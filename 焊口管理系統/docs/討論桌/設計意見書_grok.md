# 焊口/ISO 管理系統 — 設計意見書(by Grok)

## 1. 總體主張(三句內)

**主張：** 以 `weld_joint` 為事件樞紐、`drawing` 為空間錨點，其餘 11 張管制表全部做成「查詢視圖 + 匯出範本」，不要各自長成獨立資料島。畫面 70% 像 Excel 格子、30% 像焊口履歷卡；匯入改「合併」、禁止整案覆蓋。

**理由：** 現有 schema 骨幹已對（專案→系統→管線→圖面→spool→焊口），痛點在「40 欄一次攤開」與「重匯洗掉手動輸入」，不是缺表。使用者心智是「先找一批焊口、再填幾個欄位」，不是「開一張大表單」。

**取捨：** 短期不補齊採購/MTO/MDR 全流程；先把焊口管制表 + ISO + 請款 + NDE 做到「比 Excel 好用」，其餘用匯出接軌。

## 2. 資料模型:核心骨幹 + 怎麼用「一份資料、多種視圖」串

### 2.1 核心骨幹（實體表，必須維護）

| 層級 | 表 | 對應管制表 | 角色 |
|------|-----|-----------|------|
| 專案殼 | `project` | 跨案彙總 | 案子邊界 |
| 空間骨幹 | `system` → `pipe_line` → `drawing` → `spool` | ISO 圖號管制表、spool 表 | 「焊口在哪」 |
| **事件樞紐** | **`weld_joint`** | **焊口管制表** | **一切追蹤的錨點** |
| 主檔字典 | `weld_type`, `welder`, `wps`, `billing_period` | 型式/焊工/WPS/請款期別 | 下拉選項、計價規則 |
| 材料主檔 | `material_heat`, `filler_material` | 爐號/MTR | 爐號只登記一次 |
| 材料掛載 | `joint_material` | 焊口↔爐號 | 一焊口多筆、含 A/B 側 |
| 事件明細 | `inspection` | NDE/RT 清單 | 補焊、重檢、多方法 |
| 議題 | `joint_issue` | punch / 圖面問題 | 非焊口本體屬性 |

**主張：** 這 13 張表裡，真正要「手動維護」的只有上表；其餘管制表全部是 **VIEW 或匯出 SQL**，從骨幹投影出來。

### 2.2 十三張管制表 → 視圖對照（一份資料、多種視圖）

| # | 管制表 | 實作方式 | 具體欄位來源 |
|---|--------|---------|-------------|
| 1 | ISO 圖號管制表 | `drawing` + JOIN 焊口統計 | `serial_no`, `drawing_no`, `current_rev`, `scan_date`, COUNT(joints) |
| 2 | 焊口管制表 | 已有 `v_joint_full`，擴充即可 | 規格 + 製程 + NDE 摘要 + 請款 |
| 3 | spool 預製分段 | `spool` + 焊口聚合 | `spool_no`, `joint_count`, `welded`, SUM(db_count) |
| 4 | MTO 物料清單 | **暫不建表**；用 `fitting_requirement` 或匯出 | 從焊口 size/material 彙總 |
| 5 | 採購 PR→PO→收料 | **Phase 2**；現階段 Excel 外掛 | 不阻擋焊口主流程 |
| 6 | 爐號/MTR | `material_heat` 主檔 + `joint_material` | 焊口表只顯示 JOIN 結果 |
| 7 | NDE/RT 清單 | `inspection` 為真相；`weld_joint.nde_*` 為**快取摘要** | 清單視圖 SELECT inspection；摘要用最新一筆 TRIGGER 同步 |
| 8 | PWHT | `weld_joint.pwht_*` 欄位 | 篩選視圖：`pwht_required=1` |
| 9 | 試壓包+punch | `weld_joint.test_*` + `joint_issue` | 試壓包視圖 GROUP BY test_package |
| 10 | 焊工/WPS/PQR | `welder`, `wps` 主檔 | 焊口只存 FK，不複製鋼印文字 |
| 11 | 請款/計價 | `billing_period` + 焊口 `db_count` | 期別視圖 SUM(db_count) WHERE claim_status |
| 12 | 進度總表 | 已有 dashboard API | 按 system / type / status 聚合 |
| 13 | MDR 品保卷冊 | **匯出組裝** | 從 inspection + material + PWHT 拼 PDF 索引 |

### 2.3 該合併 / 拆分 / 補的

**主張 A — 拆分 NDE「雙寫」：** `weld_joint.nde_type/nde_date/nde_result/nde_report_no` 改為**唯讀摘要**（由 `inspection` 最新一筆或「最終結果」觸發器維護）。清單篩選、匯出仍用摘要欄（效能）；編輯只在 `inspection` 子表。

- **理由：** 補焊 2 次就會有 3 筆 RT；摘要欄與明細不一致是品保事故來源。
- **取捨：** 多一個同步機制；換來清單查詢仍可用索引、不需每次子查詢。
- **具體做法：** SQLite trigger `AFTER INSERT/UPDATE ON inspection` → 回寫 `weld_joint.nde_*` 與 `repair_count`。

**主張 B — 合併 `weld_joint.heat_no` 與 `joint_material`：** 廢除焊口表上的單一 `heat_no` 文字欄（或標 deprecated），材料追溯一律走 `joint_material`。

- **理由：** A 側、B 側、銲材本來就多筆；單欄位是歷史包袱。
- **取捨：** 舊 Excel 匯入需映射到 `joint_material`；短期可在匯入時自動建一筆 role=「爐號(匯入)」。

**主張 C — 補「欄位擁有權」元資料（輕量）：** 新增 `field_source` 表或焊口上的 `locked_fields JSON`：`{"weld_date":"ui","nde_date":"import","billing_period_id":"ui"}`。

- **理由：** 解決重匯覆蓋；比整案 DELETE 再 INSERT 便宜。
- **取捨：** 匯入邏輯變複雜；但 `merge_completion_file` 已證明「增量合併」可行，應升格為預設。

**主張 D — 識別鍵補強：** `weld_joint` 加 `UNIQUE(project_id, drawing_id, joint_no)`（drawing_id 可 NULL 時用 `COALESCE(drawing_id,0)` 變通）。圖面用既有 `UNIQUE(project_id, drawing_no)` + `serial_no` 作公司內主鍵。

- **理由：** 匯入時 `INSERT` 失敗就 `continue` 會靜默丟資料；應改成 UPSERT。
- **取捨：** 同一圖兩個同名焊口（極少）需人工議題處理。

**主張 E — 暫不補採購/MTO 表：** 用「焊口規格視圖」匯出給採購 Excel，等焊口主流程穩了再加 `purchase_order` / `mto_line`。

- **理由：** 採購鏈牽涉財務流程，與現場焊口輸入節奏不同；先做會分散 UX 投資。
- **取捨：** 採購仍雙軌一陣子；可接受，因目前痛點不在 PO。

## 3. UI 風格:落在光譜哪裡;格子與紀錄的界線

### 主張：落在「**可編輯試算表格子 (70%) + 右側/底部履歷卡 (30%)**」，偏 Excel 但加一層「狀態與警示」。

**理由：**
- 使用者已用 Excel 十年；改變成本最低的是「長得像 Excel、但比 Excel 會幫你算 DB、會擋錯、會記錄誰改的」。
- 全表單 App 適合「一筆一筆建檔」；配管現場是「今天要把這張 ISO 上 40 個焊口的完銲日填完」——批次格子才是主場景。
- 純 Excel 線上版缺：狀態色塊、下拉驗證、關聯主檔、稽核。

**取捨：** 不做 AG Grid / Handsontable 等重型元件；用原生 `<table>` + 固定表頭 + 虛擬捲動（或分頁）即可，維持輕量。

### 格子 vs 紀錄的界線

| 放格子（清單可編輯） | 放紀錄卡（點列展開，不一次全攤） |
|---------------------|--------------------------------|
| 銲口編號、圖號、尺寸、材質、型式、S/F、spool | 完整 inspection 歷程（多筆 RT） |
| 完銲日、組對日（批次填） | joint_material 多筆爐號/銲材 |
| 狀態、NDE 摘要（最新結果/待檢） | WPS 詳規、焊工證照到期 |
| 請款期別、請款狀態 | PWHT / 試壓細節 |
| 承包商、備註（短） | 稽核 log、議題 punch |

**具體畫面：**
```
┌─ 篩選列（可收合）────────────────────────────────────┐
│ 圖號/銲口搜尋 │ 系統▼ │ 狀態▼ │ 尺寸▼ │ 材質▼ │ 焊工▼ │ 完銲日 區間 │ [儲存篩選] │
├─ 焊口格子（固定左欄：圖號+銲口）──────────────────────┤
│ □ │ 0012│ISO-xxx│W-01│6"│A106│BW│S│2026-03-15│完銲│合格│2026.03│  ← 可直接改完銲日
│ ...（虛擬捲動 2.5 萬筆）                                │
├─ 選取列詳情（Tab）────────────────────────────────────┤
│ [摘要] [檢驗] [材料] [商務] [紀錄]                     │
└──────────────────────────────────────────────────────┘
```

**界線規則：**
- **清單永遠 ≤ 15 欄**（其餘用欄位選擇器顯示/隱藏，像 Excel 凍結欄）。
- **點列不跳大 Modal**；改為右側抽屜或底部面板（同一頁，減少「進出」迷失）。
- **「推進▶」保留在列上**——這是配管語言，比改下拉狀態直覺。

## 4. 可用性

### 4a 輸入(含批次)

**主張：** 輸入分三條高速公路，依「今天在做什麼事」進站，不要只有「編輯焊口」一條路。

| 情境 | 入口 | 欄位 | 具體步驟 |
|------|------|------|---------|
| 開案/改版 | 匯入 → **合併模式**（預設） | 規格欄：尺寸、材質、型式、圖號 | 選檔 → 預覽差異（新增 N / 更新 M / 跳過手動欄 K）→ 確認 |
| 現場完銲回報 | 「完成資料匯入」或格子批次 | `weld_date`, `shop_field`, `subcontractor` | 沿用 `merge_completion_file`；或篩「未完銲」→ 貼上 Excel 兩欄 → 對應 |
| 檢驗回來 | NDE 批次 | `inspection` 一筆 | 上傳 RT 清單 Excel（銲口+日期+報告）→ 自動長 inspection + 更新摘要 |
| 請款關帳 | 請款視圖批次 | `billing_period_id`, `claim_status` | 篩「完銲且合格」→ 勾選 → 指派期別 2026.03 |

**格子內輸入技巧（無框架可做到）：**
- Tab / Enter 跳下一格；Esc 取消。
- 日期欄：支援貼上 `115.03.15` 自動轉 ISO（後端已有 `roc_to_iso`，前端也應有）。
- 下拉欄：焊工、WPS、期別——鍵入鋼印前幾碼篩選（`<datalist>` 或自寫 20 行）。
- **批次貼上：** 從 Excel 複製一欄 → 貼到選取的多列同一欄（像 Excel fill down）。

**取捨：** 不做離線 PWA；假設工務室有 PC、現場用手機拍照回傳仍走 Excel 再合併——符合輕量邊界。

### 4b 可見性

**主張：** 2.5 萬筆不可能「一眼看完」；要讓使用者**一眼看到「今天該看什麼」**。

**三層可見性：**

1. **儀表板（已有，強化）** — 進專案先看 KPI + 異常：
   - 未完銲但已過預定日（需 `target_date` 或從 spool 出貨日推算）
   - RT 待檢超過 N 天（`weld_date` 有、`nde_date` 空）
   - 不合格未關閉（`nde_result=不合格` 且 `repair_count` 未對應新合格檢驗）
   - 已完銲未掛期別（請款漏網）

2. **清單預設視圖（Saved Views）** — 取代「顯示前 300」：
   - 內建：`我的待辦`（狀態∈組對/完銲/待檢）、`本週完銲`、`RT 不合格`、`未請款`
   - 每個視圖 = 一組篩選 + 欄位集合 + 排序
   - 列數顯示「共 25,312 筆｜符合 847 筆｜顯示 1–200」

3. **列上視覺編碼** — 不靠多欄：
   - 狀態色條（已有 badge，左側加 4px 色條更醒目）
   - NDE：綠/紅/灰三角
   - 請款：期別碼或空心圓（未掛）

**技術：** 後端 `list_joints` 已支援 `limit/offset/total`；前端改 **分頁 200 + 無限捲動載入**，移除硬編 300。2.5 萬筆 DOM 全渲染會卡——只渲染視窗內 ~50 行（虛擬捲動，純 JS 約 150 行）。

**取捨：** 虛擬捲動實作成本 > 分頁按鈕；但配管人員討厭按頁碼，建議無限捲動 + 跳至銲口號。

### 4c 篩選

**主張：** 篩選是此系統的**主武器**；應做成「Excel 自動篩選 + 儲存常用條件」，後端 API 其實已有一半（`billing`, `spool_id`, `drawing_id` 已有，前端沒接）。

**篩選面板欄位（建議全接 API）：**

| 類別 | 欄位 | UI 元件 | SQL 條件範例 |
|------|------|---------|-------------|
| 識別 | 圖號/銲口/流水號 | 搜尋框（已有） | LIKE |
| 階層 | 系統、管線、圖面、spool | 級聯下拉 | FK / JOIN |
| 規格 | 尺寸、材質、型式、S/F、分類 | 多選 checkbox | IN (...) |
| 製程 | 打底/蓋面焊工、WPS | 多選 | welder_root_id IN |
| 進度 | 狀態（多選）、完銲日區間、組對日區間 | 日期 range | BETWEEN |
| 檢驗 | NDE 方式、結果、待檢（有 type 無 date） | 三態：全部/是/否 | IS NULL 組合 |
| 商務 | 請款期別、請款狀態、承包商 | 下拉 | 已有 billing 參數 |

**進階（Phase 2）：** 條件列建構器「且/或」——初期用 **多個 AND 就夠**（配管查詢 95% 是 AND）。

**具體 UX：**
- 篩選列預設收合，只露搜尋 + 系統 + 狀態 +「更多篩選」。
- 套用後 URL 帶 query（可書籤、可貼給同事：「這批 RT 不合格」）。
- 顯示 **作用中篩選標籤**（可點 × 移除），像電商 facet。

**取捨：** 不做法條件全文搜尋；焊口資料結構化程度高，精確篩選比模糊搜尋有用。

### 4d 一個焊口太多事件

**主張：** 焊口一生用 **「時間軸 + 分頁 Tab」**，禁止 40 欄單頁表單；清單只看摘要，細節分域。

**焊口詳情結構（取代現有 `jointFields()` 一次全開）：**

```
標題列: W-01 ｜ ISO-CP129-0012 ｜ 6" BW A106 ｜ [完銲] ｜ RT合格 ｜ 2026.03未請款
快速動作: [推進狀態▶] [加一筆檢驗] [掛爐號] [指派期別]

Tab 1 摘要（唯讀多、可改少）: 規格 6 欄 + 製程 6 欄 — 一屏看完
Tab 2 檢驗: inspection 表格 + 「＋新增」 — 每行一事件，含補焊次序
Tab 3 材料: joint_material — A側/B側/銲材分組
Tab 4 商務: 期別、claim、承包商、DB數（可手動覆寫需留痕）
Tab 5 歷程: audit_log 篩選此焊口 — 誰何時改了完銲日

（摺疊區）PWHT / 試壓 / 備註 — 預設收起，有值才自動展開
```

**時間軸（可選增強）：** 組對日 → 完銲日 → RT#1(不合格) → 補焊 → RT#2(合格) → 掛期別——從 `inspection` + `audit_log` 自動生成，給品保看故事。

**狀態機：** 保留「推進▶」；狀態轉換時**只開該階段必填欄**（組對→只要 fitup_date；完銲→weld_date；待檢→提示加 inspection）。

**取捨：** Tab 比單頁捲動多一次點擊；但認知負荷從 40 欄降到每次 6–8 欄，淨效益為正。

## 5. 最該先決的考量

### 5.1 單一真實來源 + 匯入不洗資料（最關鍵）

**現況問題：** `import_weld_control(..., replace=True)` 預設 `DELETE FROM project`，整案連手動輸入的 `weld_date`、期別、inspection 全滅。這是使用者不敢用系統當主檔的根因。

**主張：匯入預設改「合併 UPSERT + 欄位擁有權」**，整案覆蓋改為需二次確認的「重建專案（危險）」。

**欄位擁有權規則（建議）：**

| 來源 | 可覆寫欄位 | 永不覆寫（除非勾選強制） |
|------|-----------|------------------------|
| Excel 焊口管制表（初匯） | 規格：size, material, weld_type, db_count, drawing 對應 | — |
| Excel 完成資料 | weld_date, shop_field, subcontractor（僅當 DB 為空） | 已有 weld_date 的列 |
| UI 手動 | 一切曾編輯過的欄位 | import 跳過 |
| inspection 子表 | 永不從 Excel 摘要欄覆蓋已有 inspection 列 | 改走獨立 NDE 匯入 |

**具體實作步驟：**
1. 匯入前：選模式「新增合併（預設）/ 僅新增 / 重建（危險）」。
2. 合併鍵：`project_id + drawing_no + joint_no`（與 UNIQUE 一致）。
3. UPSERT：`INSERT ... ON CONFLICT DO UPDATE SET col=excluded.col WHERE field_source.col IS NULL OR field_source.col='import'`。
4. 匯入後出 **差異報告**：「更新 120、跳過手動保護 38、新增 15、圖面新增 2、警告：3 銲口找不到圖號」。
5. 推廣既有 `merge_completion_file` 為 UI 一級按鈕「匯入完成資料」。

**取捨：** 匯入程式變長；但這是「系統能不能取代 Excel」的生死線，優先於任何 UI 美化。

### 5.2 識別碼規範

**主張：三層 ID，各管各的：**

| 層級 | 業務鍵 | 系統鍵 | 規範 |
|------|--------|--------|------|
| 圖面 | `drawing_no`（業主圖號，可改版） | `serial_no`（公司流水，不變） | 匯入/匯出/請款用 serial；對業主溝通用 drawing_no |
| 焊口 | `joint_no`（圖內編號） | `id`（SQLite PK） | 業務唯一 = `(project, drawing, joint_no)` |
| 跨表引用 | — | FK integer | UI 永遠顯示業務鍵，存檔用 FK |

**取捨：** 不引入 UUID；單機 SQLite 整數 PK 夠用，人也能記「#48291」。

### 5.3 版次歷史

**主張：圖面版次與焊口版次分開管。**

- **圖面：** 已有 `drawing_revision`；改版時新增 revision 列、更新 `drawing.current_rev`；**焊口不刪**，舊焊口掛 `drawing_id`，若圖號廢止則 drawing.status=作廢。
- **焊口規格變更：** 不覆寫歷史；用 `audit_log` + 可選 `weld_joint_rev`（僅在「業主要求留版次快照」時加，初期可不做）。
- **匯入新版 Excel：** 規格變更寫 audit「size 4"→6"」；若已完銲則標 `joint_issue` 待確認，不自動改。

**取捨：** 完整焊口版次表成本高；品保真正要的是 inspection + 材料不可改（改要留 log），規格欄用 audit 即可。

### 5.4 備份安全

**主張：SQLite 單檔的優勢就是「複製即備份」——做成使用者看得懂的習慣，不是隱藏 DBA 功能。**

**具體做法：**
1. **每次匯入前自動備份：** `weld.db` → `backups/weld_YYYYMMDD_HHMMSS_匯入前.db`（保留最近 10 份）。
2. **run.bat 同目錄顯示「上次備份時間」**；設定頁一鍵「立即備份」「還原（需確認碼）」。
3. **匯出 = 邏輯備份：** 整案匯出 xlsx（多 sheet 對應視圖），給人員習慣性存檔。
4. **WAL 模式 + 正常關閉：** `PRAGMA journal_mode=WAL`；避免複製到半寫入檔（備份前 `CHECKPOINT`）。

**取捨：** 不做雲端同步；多台共用靠「專用資料夾 + 約定誰開機」或定期複製 db——符合少數人單機假設。

## 6. 只能先做 3 件事

| 優先 | 做什麼 | 為什麼 | 具體交付物 |
|------|--------|--------|-----------|
| **#1** | **匯入改合併 + 欄位保護 + 匯入差異報告** | 不解決「重匯洗掉」= 系統永遠是 Excel 附屬品；這是信任基礎 | `replace=False` 預設、UPSERT、UI 三模式、差異摘要 Modal |

## 7. 我最想強調的一點 + 我可能錯在哪(自我打臉一次)

### 最想強調

**系統的產品定義應該是「焊口管制表的線上版，Excel 是 IO 埠」——不是「把 13 張表都搬進來的 ERP」。**  
所有設計判斷問一句：「工務明天下午三點要關 2026.03 請款，他能不能在五分鐘內篩出『完銲+RT合格+未掛期別』、批次掛期別、匯出請款表？」做不到，其他功能都是噪音。  
因此 **#1 匯入合併** 與 **#2 篩選** 比任何新表都重要。

### 我可能錯在哪

**我可能高估了「格子內直接編輯」對非工程師的友善度。**  
配管老手在 Excel 可以狂按 Tab；但在瀏覽器 `<table>` 裡，沒有框架支援時，貼上、Undo、填滿向下常常卡頓或行為不一致，反而比「選多筆 → 按鈕批次填日期」更難教。  
若實地測試發現使用者寧可匯入小 Excel 片段也不想在格子上改，就應退守 **「唯讀格子 + 強批次面板」**（例如：篩出 200 筆 → 一個日期欄套用全部），把「像 Excel」放在匯入/匯出與欄位排列，而不是逐格編輯——那條路實作更簡單、教學成本更低，我上面 70% 可編輯格子的主張就要調降為 40%。
