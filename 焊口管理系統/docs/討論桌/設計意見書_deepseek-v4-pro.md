# 焊口/ISO 管理系統 — 設計意見書（by deepseek-v4-pro）

## 1. 總體主張（三句內）

**把 Excel 最好的部分（格子、篩選、一眼看穿）和 App 最好的部分（結構、引導、不讓你亂填）各取所長，焊口清單用可篩可排的格子、焊口細節用時間軸+摺疊區塊，兩者之間用「點一下」切換。** 當前最致命的三件事依序是：匯入會洗掉手動輸入→改為合併；篩選只有三項→擴到十五項任意組合；焊口表單一次攤開 40 欄→改為分區摺疊+時間軸。

---

## 2. 資料模型：核心骨幹 + 怎麼用「一份資料、多種視圖」串

### 主張：以 weld_joint 為唯一心臟，畫出「主骨幹 5 表 + 事件 5 表 + 參考 4 表」的分層架構

**理由：** 目前 17 張表已經把該有的都建了，問題不在少表，在於：
1. `weld_joint` 身上同時背了「當前摘要值」（nde_result, nde_date, nde_report_no）和「事件明細」（inspection 表），等於同一件事存兩份，遲早不一致；
2. `drawing` 和 `weld_joint` 之間少了「線別（pipe_line）」的強制路徑，導致部分焊口直接掛圖面而略過管線；
3. 缺「試壓包」獨立表——試壓包是現場施工的關鍵管理單元，一個試壓包跨多圖多焊口，目前只存在焊口上的 `test_package` 字串欄位，無法反查「這個試壓包裡有哪些焊口」。

**具體做法（分層）：**

```
主骨幹 5 表（逐層掛載，缺一不可）：
  project → system → pipe_line → drawing → weld_joint
                                          └→ spool（一張圖可有多個 spool，spool 是焊口的可選容器）

事件 5 表（發生在焊口身上的時間序列事件）：
  inspection（NDE 每次檢驗，一焊口可多次）
  pwht_event（獨立 PWHT 紀錄，一焊口可能多次升溫）
  test_event（試壓事件，掛到試壓包）
  joint_issue（問題/異常）
  joint_material（材料追溯，多筆）

參考 4 表（被焊口外鍵參照，不直接存事件）：
  welder / wps / weld_type / billing_period
  + material_heat / filler_material（被 joint_material 參照）
```

**該合併/拆分/補的：**

| 動作 | 對象 | 理由 |
|------|------|------|
| **拆** | `weld_joint` 上的 NDE 摘要欄位（nde_type/nde_date/nde_result/nde_report_no/repair_count）→ 全部由 `inspection` 表推導（VIEW 或查詢時 MAX/最新一筆） | 現在 inspection 新增時會回寫 weld_joint（app.py:530-534），這就是同一件事存兩份的原罪。焊口本身不該存 NDE 摘要，應由事件表推導。 |
| **補** | `test_package` 獨立表（test_package: id, project_id, code, test_type, test_medium, design_pressure, test_pressure, test_date, result, remark） | 試壓包是現場管理的重要單位，跨焊口，不能只是一個字串欄。焊口的 `test_package` 欄改為 FK 指向此表。 |
| **補** | `pwht_event` 獨立表（pwht_event: id, joint_id, pwht_date, curve_path, result, remark）取代焊口上的 pwht_required/pwht_done/pwht_date | PWHT 是一次或多次事件，不是布林。 |
| **留但約束** | `weld_joint.line_id` 應設為 NOT NULL（或至少在匯入時強制透過 drawing→pipe_line 自動補） | 目前可為 NULL，導致以系統篩選時透過 pipe_line JOIN system 的邏輯脆弱。 |
| **補索引** | `weld_joint(size), weld_joint(material), weld_joint(shop_field), weld_joint(weld_date), weld_joint(welder_root_id), weld_joint(welder_cap_id)` | 目前只建了 project_id, drawing_id, status, billing_period_id 四個索引。尺寸/材質/S·F/焊工/日期是使用者最想篩的維度，沒有索引 2.5 萬筆會慢到不能用。 |

### 取捨

- **取：** 犧牲一點「焊口表本身就有 NDE 摘要」的便利性，換取資料不重複、不怕不一致。
- **捨：** 不做完全正規化（例如尺寸/材質/SCH 在焊口、管線、圖面都可能重複出現——這些是 Excel 時代的冗餘，但現場人員習慣在同一列看到所有規格，移除反而增加 JOIN 次數和使用者困惑）。**刻意保留合理冗餘**。
- **「一份資料、多種視圖」的實現：** 用 SQLite VIEW 產生「焊口管制表視圖」「NDE 清單視圖」「請款彙總視圖」「試壓包視圖」，每個 VIEW 就是那 13 張管制表的一張。同一個焊口在不同 VIEW 出現，但資料只在 weld_joint + inspection + materials 等基礎表裡各存一份。

## 3. UI 風格：落在光譜哪裡；格子與紀錄的界線

### 主張：清單用「試算表式格子」，單筆用「分區摺疊+時間軸紀錄」，界線畫在「是不是在看一筆的完整一生」

**理由：**「純 Excel 式」無結構、「純表單式」批次慢——目前的設計文件已經指出混合路線是對的，但我認為現有實作偏向右端太遠了（點一個焊口跳出 40 欄 flat form）。具體光譜落點：

```
純 Excel ──── 試算表格子 ────★我們該在這裡──── 表單/紀錄 App
（巨集）      （Airtable 式）  │                （Salesforce 式）
                               │
                         清單=格子
                         單筆=摺疊+時間軸
```

**界線畫法（具體）：**

| 畫面 | 用什麼 | 原因 |
|------|--------|------|
| 焊口清單 | **格子(grid)**——12~15 欄、可排序、可多選、可內聯編輯（inline edit）特定欄位（狀態、完銲日、焊工、NDE 結果） | 使用者需要在 2.5 萬筆中快速掃描、比對、批次改。格子是唯一能「一眼看穿」的形式 |
| 圖面清單 | 格子，8~10 欄 | 同上，量較小 |
| Spool 清單 | 格子 | 同上 |
| 焊口細節（點開後） | **紀錄式——分區摺疊 + 時間軸**（見 4d） | 一個焊口的一生是「事件序列」，不是「一張大表」 |
| 儀表板 | 卡片 + 長條圖（已做好） | ✅ 維持 |
| 基礎資料維護 | 精簡表格 + modal 編輯（已做好） | ✅ 維持 |
| 匯入/匯出 | 面板（已做好） | ✅ 維持 |
| 請款期別 | 格子+內聯編輯 | 同清單級 |

**具體畫面配置：**
焊口清單格子的預設 13 欄（按現場人員眼睛掃描順序）：流水號、圖號、**銲口編號**、尺寸、SCH、材質、型式、S/F、Spool、完銲日、狀態（彩色圓點）、NDE結果（彩色圓點）、請款期別。每欄寬度自適應，最後一欄固定「▶ 詳情」按鈕。

### 取捨

- **取：** 清單資訊密度高（像 Excel），犧牲手機友善度——現場人員用筆電/桌機居多，不需要 responsive 到手機。
- **捨：** 不做 Google Sheet 那種任意格自由打字——那不是結構化資料，是災難。格子內的編輯只開放給「狀態推進」「完銲日」「焊工」等少數高頻欄位。

## 4. 可用性

### 4a 輸入（含批次）

**主張：三層輸入策略——匯入建骨架、格子批次補肉、表單單筆精修**

**具體做法：**

**第 1 層：Excel 匯入（建骨架）——最該先修**
- 目前 `import_weld_control(replace=True)` 刪掉整個專案重來（importer.py:219-221）→ **這是 SSOT 的原罪**。改為：
  - 以 `(drawing_no, joint_no)` 為業務自然鍵做 **upsert**：存在就更新（但只更新「匯入來源有的欄位」，不蓋手動已填的值），不存在就 insert。
  - 手動欄位白名單：`welder_root_id, welder_cap_id, fitup_by, fitup_date, pwht_*, test_*, billing_period_id, claim_status, remark`——這些欄位匯入時永不覆蓋，只由系統 UI 手動填寫。
  - 匯入後回報：新增 X 筆、更新 Y 筆、略過 Z 筆（已存在且無變更）。

**第 2 層：格子批次編輯（補肉）**
- 在焊口清單格子上，勾選多筆（checkbox 第一欄）→ 右鍵或工具列出現「批次指派」：
  - 批次設狀態、批次設完銲日、批次設焊工、批次歸 spool、批次歸請款期別。
  - 一次操作影響數百筆，這是 Excel 做不到（或要做得很痛苦）、系統最能發揮的場景。
- 實作：用 `fetch` 一次送 `{joint_ids: [...], field: "status", value: "完成"}`，後端一個 UPDATE WHERE id IN (...) 搞定。

**第 3 層：表單單筆精修**
- 見 4d。

**快捷輸入輔助：**
- 焊口編號自動建議：選了圖面後，根據圖面上已有焊口編號最大值 +1 建議（例如圖上已有 W01~W12，新增時預填 W13）。
- 完銲日期 = 今天（一個按鈕「今天」）。
- 狀態推進一鍵（已做 `advance` API ✅），但建議在清單格子上直接做（不用點進去）。

### 取捨

- **取：** 匯入只建骨架、不覆蓋手動值，代價是匯入邏輯變複雜（需要逐筆比對），但這是信任的基礎——使用者必須相信「我不會因為重新匯入就弄丟上週填的焊工名字」。
- **捨：** 不做 Excel 般的自由公式格（`=A1+B1`），那是表格軟體的事。

### 4b 可見性（2.5 萬筆怎麼一眼看到重點）

**主張：虛擬捲動 + 摘要列 + 條件式色標，讓使用者在清單上就能判斷狀況，不必點進去**

**具體做法：**

1. **虛擬捲動（virtual scroll）取代 LIMIT 300：**
   - 目前 `limit=300` 且提示「共 24,879 筆（顯示前 300）」根本看不到後面 24,579 筆。
   - 做法：前端只渲染可視區 + 上下各一屏的 DOM 節點（約 60~80 個 `<tr>`），捲動時動態替換內容。對 vanilla JS 來說，用 `IntersectionObserver` + `scrollTop` 計算即可，約 100 行程式，不需要引入框架。
   - 後端 API 改為支援 `offset`（已有）但前端的 limit 設為可視區大小而非 300——實際上，前端虛擬捲動需要的是「全部 ID 清單」或「分頁載入」，建議用**游標分頁**：`GET /joints?cursor=12345&limit=200`，每次拿 200 筆，前端累積顯示。

2. **顏色一眼看出問題：**
   - 狀態欄：彩色圓點（已做 ✅）
   - NDE 結果：綠色（合格）/ 紅色（不合格）/ 灰色（未檢）/ 黃色（待檢中）
   - 完銲日期超過 N 天還沒 NDE：整列淡紅色背景（滯留預警）
   - 焊口逾期未請款：金額欄黃底

3. **清單頂部摘要列（凍結）：**
   ```
   篩選結果：1,847 口 ｜ 已完成 1,203 (65%) ｜ DB 數合計 4,521.5 ｜ RT 合格率 97.3% ｜ 不合格 12 口
   ```
   這一列隨篩選條件即時更新，讓使用者知道「我現在看的是什麼範圍、狀況如何」。

4. **欄位顯示切換：**
   - 工具列上放一組「顯示欄位」多選下拉：勾選想看哪些欄，預設 13 欄全開但可縮到 6 欄核心（銲口編號、圖號、尺寸、材質、狀態、NDE）給小螢幕。

### 取捨

- **取：** 花工程做虛擬捲動（約 100~150 行 vanilla JS + 後端游標分頁），換取「能看到全部 2.5 萬筆」的基本可用性。
- **捨：** 不做全文搜尋引擎（Elasticsearch 等），SQLite 的 LIKE + 索引對 2.5 萬筆綽綽有餘。

### 4c 篩選（怎麼任意條件精確篩選）

**主張：擴充到 12~15 個篩選維度，以「篩選列（filter bar）」形式橫排在清單上方，支援自由組合、即時生效**

**具體做法：**

目前的篩選列：
```
[搜尋框] [狀態▼] [系統▼]
```
只有 3 個。

改為兩排篩選列（可收合，預設顯示第一排 8 個）：

**第一排（高頻）：**
```
[搜尋框(銲口/圖號/流水號)]
[狀態▼: 規劃/組對/完銲/待檢/合格/不合格/試壓/完成]
[系統▼: AI/AP/FF/...]
[S/F▼: S 預製 / F 現場]
[尺寸▼: 1/2" / 3/4" / 1" / 1.5" / ... / 24"]
[材質▼: A106 / A312 / ...]
[NDE結果▼: 合格/不合格/未檢]
[請款期別▼: 2026.01 / 2026.02 / ...]
```

**第二排（展開後）：**
```
[銲接型式▼: BW/SW/IW/FW/...]
[焊工▼: 鋼印列表]
[WPS▼: WPS 編號列表]
[完銲日期區間: 起～迄]
[NDE 日期區間: 起～迄]
[分類▼: 消防/工業級/...]
[承包商▼]
```

**實作細節：**
- 每個下拉的選項來自 `lookups` API 並附加 COUNT（例如「A106 (1,847)」），讓使用者知道選了會剩多少筆。
- 多選支援：狀態、系統、材質、型式、NDE 結果應支援多選（checkbox dropdown 或按住 Ctrl 多選）。例如想看「A106 + A312，狀態=完銲或待檢」。
- 篩選條件即時反映在 URL hash（例如 `#status=完銲,待檢&material=A106&system=AP`），可複製分享給同事、重整後不丟失。
- 後端 API 已有雛形（`list_joints` 支援 q/status/system/drawing_id/billing/spool_id），只需擴充參數，並用動態 SQL 組裝（注意 SQL injection，用參數化查詢）。

**效能注意：** 每個篩選維度都需要對應的資料庫索引（見第 2 節補索引建議）。沒有索引的篩選在 2.5 萬筆下會全表掃描，勉強可接受但累積多條件會變慢。

### 取捨

- **取：** 12~15 個篩選器看起來很多，但現場工務每天要用，花 10 分鐘學一次就值回票價。介面用兩排+收合避免擁擠。
- **捨：** 不做自然語言查詢（例如「幫我找 A106 材質上個月完銲但未檢驗的焊口」），那是 LLM 的事，不是 UI 的事。結構化篩選器對結構化資料才是最準的。

### 4d 一個焊口複雜的一生怎麼呈現才不難受

**主張：時間軸為主、摺疊區塊為輔、只顯示「現在該填什麼」**

**放棄現有做法：** 一個 modal 攤開 40 個欄位，分成 5 個 section（基本資料 / 製程追溯 / NDE / PWHT試壓 / 商務），但仍然是 flat form——對一個已經完銲+已檢驗+已請款的焊口來說，等於強迫使用者看 40 個已經過去的欄位。

**改用「時間軸 + 分區摺疊」佈局：**

```
┌──────────────────────────────────────────────┐
│ 焊口 W045 — 圖號 ISO-A12-003 — S 預製        │
│ 尺寸 6" × SCH40 × A106 ｜ BW 對焊 ｜ DB: 6.0 │
│ 狀態：● 完成                                  │
├──────────────────────────────────────────────┤
│                                              │
│  ● 生命時間軸                                │
│  ┌──────────────────────────────────────┐    │
│  │ 2026-01-15  規劃建立                  │    │
│  │ 2026-02-03  組對 (組對者: 陳)         │    │
│  │ 2026-02-05  完銲 (打底: 王○ / 蓋面: 李○)│   │
│  │ 2026-02-10  RT 檢驗 → 合格 (報告 R-089)│   │
│  │ 2026-03-01  歸入試壓包 TP-07          │    │
│  │ 2026-03-15  試壓通過                  │    │
│  │ 2026-03-20  請款 (期別 2026.03, 已請款)│   │
│  └──────────────────────────────────────┘    │
│                                              │
│  ▶ 規格詳情 (摺疊)                           │
│  ▶ 製程與焊工 (摺疊)                         │
│  ▶ NDE 檢驗紀錄 (2 筆) (摺疊)               │
│  ▶ 材料追溯 A側/B側/銲材 (摺疊)             │
│  ▶ 試壓紀錄 (摺疊)                           │
│  ▶ 商務/請款 (摺疊)                          │
│  ▶ 問題紀錄 (摺疊)                           │
│                                              │
│  [+ 新增事件 ▼]  (下拉: 檢驗/材料/問題/...)  │
│  [推進狀態 ▶]                                │
└──────────────────────────────────────────────┘
```

**設計原理：**
1. **頭部固定摘要**（焊口身分證+目前狀態）——永遠可見，5 秒內知道這是哪一口。
2. **時間軸是主角**——一個焊口的生命是事件序列，不是屬性清單。每個事件自動從各子表（inspection, joint_material, test_event 等）彙整，依時間排序。時間軸上的事件是「已發生的」，一眼看完一生。
3. **摺疊區塊是配角**——當使用者要編輯特定區塊的細節時再展開。預設全部摺疊，只有時間軸展開。
4. **「新增事件」下拉按鈕**——引導使用者「下一步做什麼」：要加檢驗？加材料？記錄試壓？而不是攤開 40 欄讓他自己找。
5. **「推進狀態」按鈕醒目**——因為這是最常用的操作（已做 ✅）。

**時間軸的資料來源：**
- 從 `audit_log`（建立日）+ `weld_joint`（fitup_date, weld_date）+ `inspection`（inspect_date, result）+ `test_event`（test_date）+ `billing_period`（歸入日）自動彙整。
- 不需要手動維護時間軸——它是查詢結果，不是一張表。

### 取捨

- **取：** 時間軸這種「閱讀優先」的佈局，犧牲了一點編輯效率（要多點幾下展開），但大幅提升理解效率。使用者 90% 時間在「看」焊口狀態，10% 時間在「改」。
- **捨：** 不做 Gantt 圖/進度圖——那是進度總表的事（第 12 張表），不是單一焊口該有的。

## 5. 最該先決的考量

### 5.1 單一真實來源（SSOT）——匯入覆蓋問題 ★最優先★

**現況：** `import_weld_control(replace=True)` 會 `DELETE FROM project WHERE id=?`（importer.py:220），整案砍掉重來。這等於宣告「每次匯入 Excel，你手動補的所有東西（焊工、NDE 結果、材料追溯、試壓）全部歸零」。

**主張：匯入永遠只做 upsert + 白名單保護，永不刪除**

**具體規則：**
1. 以 `(project_id, drawing_no)` 為圖面自然鍵、`(project_id, drawing_id, joint_no)` 為焊口自然鍵，做 INSERT OR UPDATE（SQLite 的 `INSERT ... ON CONFLICT DO UPDATE`）。
2. 匯入來源有的欄位才更新；手動欄位白名單（`welder_root_id, welder_cap_id, fitup_by, fitup_date, pwht_*, test_*, billing_period_id, claim_status, remark, nde_*`）**永不**被匯入覆蓋。或者更激進：匯入只寫入「規格層」欄位（尺寸/厚度/SCH/材質/型式/分類/DB/S·F），其餘全部視為手動域。
3. 匯入報告清楚列出：新增 X 筆、更新規格 Y 筆、手動域保留 Z 筆。
4. 若使用者真的需要「整案重來」，提供獨立的「清除專案資料」按鈕（帶確認 dialog），而不是藏在匯入流程裡。

**更進一步：** 引入「資料來源」標記——每個焊口記錄它是「從 Excel 匯入」還是「手動建立」，方便稽核。

### 5.2 識別碼規範化

**現況：** 匯入時只做簡單 strip（importer.py 的 `normalize`），但 se-39 合併完成資料時發現「流水號 vs 圖號」對不上的問題（README:99-100）。

**主張：匯入時強制規範化所有識別碼，以規則引擎處理**

**規則：**
1. 焊口編號：trim → 全形數字轉半形 → 移除所有空白 → 統一大寫。例如「Ｗ０４５」→「W045」。
2. 圖號：trim → 全形轉半形 → 統一連接符（「-」「–」「—」都變「-」）→ 移除副檔名（若有人把 `.pdf` 寫進去）。
3. 流水號：強制整數（去掉小數點、補零不一致等）。
4. 系統代碼：大寫 + trim（AI, AP, FF 等）。
5. 規範化後的結果存入 `joint_no_normalized`、`drawing_no_normalized` 欄位（或在查詢時即時正規化），原始值保留在原始欄位供參考。

### 5.3 版次歷史

**現況：** `drawing` 有 `current_rev` 欄，`drawing_revision` 表存在但幾乎未使用。

**主張：不需要完整的 temporal database，只需要「圖面版次紀錄 + 焊口變更稽核」**

**理由：** 完整版次系統（每次修改都留一版）對這個規模和使用場景過重。真正的需求是：
1. 能回答「這張圖現在是哪一版」
2. 能回答「這個焊口什麼時候被誰改成不合格」

**做法：**
- `drawing_revision` 表維持，每次改版 append 一筆（已建 ✅）
- `audit_log` 已記錄每筆 UPDATE（已建 ✅）——這對焊口級的追溯已足夠
- 不建議焊口做 soft delete + versioning——焊口不是文件，是物理存在，不會「改版」，只會被修正

### 5.4 備份安全

**主張：三個機制並行——自動備份 + 手動快照 + DB 不入 git**

1. **自動備份：** app 關閉時（或每天首次啟動時）複製 `weld.db` → `weld_backup_2026-06-30.db`，保留最近 7 天的每日備份。
2. **手動快照：** 匯入/大量修改前，UI 提示「建議先備份」並提供一鍵備份按鈕。
3. **DB 不入 git：** `.gitignore` 已有 `weld.db`？檢查。應只進結構（schema.sql），不進資料。
4. **匯出備份：** 定期匯出全案焊口為 Excel 存檔——不是為了再用，是為了災難時人類可讀。

### 取捨

- **取：** SSOT 和識別碼是地基。地基不穩，上面蓋什麼都會塌。
- **捨：** 不做完整的版次系統（保留所有歷史版本）——代價太高，現有 audit_log 對焊口級追溯已足夠。

## 6. 只能先做 3 件事

### 第 1 件：匯入改為「合併模式」，不洗掉手動資料 ★最優先★

**為什麼：** 這是信任問題。現在只要重新匯入 Excel，所有手動輸入（焊工、檢驗、材料、試壓）全部消失。使用者在「系統」和「Excel」之間永遠選 Excel，因為系統會吃掉他的工。修好這條，系統才能成為真相來源。

**怎麼做：**
1. `importer.py` 的 `replace=True` 改為 `replace=False`，`DELETE` 邏輯移除
2. 焊口以 `(project_id, drawing_id, joint_no)` 為鍵做 upsert
3. 手動域白名單保護
4. 匯入結果回報清楚（新增/更新/略過）
5. 增加 `merge-completion` 的穩健性（目前已有雛形，見 `merge_completion_file`）

**驗收標準：** 匯入同一份 Excel 兩次，第二次回報「更新 0 筆，略過 X 筆」；手動在 UI 改了某焊口的焊工，再匯入同一份 Excel，焊工不被覆蓋。

### 第 2 件：擴充篩選器到 12 維度 + 虛擬捲動清單

**為什麼：** 目前「只能看前 300 筆 + 只能搜尋/狀態/系統三種篩選」等於系統對 se-39（24,879 筆焊口）根本無法使用。使用者找不到他想找的焊口，系統就形同虛設。

**怎麼做：**
1. 後端 API：擴充 `list_joints` 的查詢參數（size, material, weld_type, shop_field, nde_result, welder, date_from, date_to, spool_id, billing_period_id），全部參數化
2. 補資料庫索引（size, material, shop_field, welder_root_id, welder_cap_id, weld_date）
3. 前端：擴充篩選列到兩排、下拉來自 lookups + COUNT
4. 前端：虛擬捲動（或至少做到分頁載入，offset/limit 介面已有，前端補頁碼或無限捲動）
5. 篩選狀態存入 URL hash

**驗收標準：** 能篩出「系統=AP、材質=A106、S/F=S、狀態=不合格」的所有焊口，並看到完整清單、摘要統計。從 24,879 筆中縮到 37 筆，3 秒內完成。

### 第 3 件：焊口細節頁改為「時間軸 + 摺疊分區」

**為什麼：** 這是「一個焊口 40 欄」痛點的直接解法。使用者每天要看好幾十個焊口的狀態，不是要填 40 欄，而是要一眼看到「這個焊口走到哪了、上次發生什麼事、下一步要做什麼」。

**怎麼做：**
1. 焊口 modal 重新設計（見 4d 的 ASCII mockup）
2. 頂部固定：焊口身分證摘要 + 狀態徽章
3. 主體：時間軸（從 audit_log + 各事件表彙整，依時間倒序）
4. 下方：摺疊區塊（規格/製程/NDE/材料/試壓/商務/問題），預設全部摺疊
5. 底部：推進狀態按鈕 + 新增事件下拉
6. 保留現有 modal 機制（overlay + popup），只改內容

**驗收標準：** 打開一個有完整生命週期的焊口（DEMO 專案），5 秒內能在時間軸上看完它的一生；要編輯焊工時，展開製程摺疊區、改完儲存，不需要在 40 個欄位中找。

## 7. 我最想強調的一點 + 我可能錯在哪（自我打臉一次）

### 我最想強調的一點

**「匯入不覆蓋手動值」是所有設計決策的基石，沒有這個，其他都是沙上城堡。** 現場人員的信任只有一次機會：如果他發現自己花了三小時輸入的焊工名字因為同事重新匯入 Excel 就消失，他永遠不會再用這個系統。這不是技術問題，是信任問題。技術上這只是一個 `replace=False` + 白名單的改動，但心理上這是「系統從敵人變成夥伴」的轉折點。

### 我可能錯在哪

**我可能高估了「時間軸」在沒有足夠事件資料時的價值。** 時間軸的威力來自事件豐富——如果一個專案剛匯入，大部分焊口只有「規劃建立」一條時間軸，那它就只是一條線，沒有比現有的 status badge 更好用。在事件資料稀疏的初期，摺疊表單可能更務實。

**修正方案：** 不要一次就全面改成時間軸。先用「分區摺疊表單」（把現有 40 欄按 section 摺疊，預設只展第一區）作為 MVP，等事件資料夠豐富（inspection 有資料、材料有掛、試壓有紀錄）再升級成時間軸。也就是說，第 3 件優先事項的實作可以分兩階段：
- Phase A（立即）：分區摺疊表單
- Phase B（等資料夠豐富）：時間軸化
