# 焊口/ISO 管理系統 — 設計意見書(by ChatGPT 5.5)

## 1. 總體主張(三句內)

**主張:**  
系統不要再把「焊口」當成一張 40 欄大表單，而要把它當成一個有身分、有狀態、有事件履歷的工作物件。  
SQLite 是唯一真相，Excel 只能是匯入/匯出與對帳格式；畫面則應提供多個「像 Excel 的工作視圖」。  
優先解決資料不被重匯洗掉、焊口識別碼穩定、篩選可精確組合，這三件事比新增功能更重要。

**理由:**  
目前最大的痛不是資料不夠，而是資料都攤在同一層，使用者找不到、改不安心、也不知道目前該處理哪一批。

**取捨:**  
不追求一次做完整 ERP；先把 ISO、焊口、NDE、請款、材料追溯這幾條主流程穩住。

**具體做法:**  
用「主清單 + 篩選器 + 右側焊口摘要 + 分頁事件紀錄」取代 40 欄長表單。

## 2. 資料模型:核心骨幹 + 怎麼用「一份資料、多種視圖」串

**主張:**  
13 張管制表不應全部變成 13 張互相複製的資料表，而應拆成「核心骨幹表、事件表、參照主檔、輸出視圖」四層。

**理由:**  
現場真正需要的是同一批焊口資料可以被不同角色看成不同表：工務看進度、品管看 NDE、材料看爐號、商務看請款。這些不該各存一份。

**取捨:**  
資料模型會比單一大表多幾張表，但查詢、匯出、版次、追溯會穩很多。SQLite 完全足夠支撐 2.5 萬焊口，問題不在資料庫大小，而在設計邊界。

**具體做法:**  

核心骨幹建議如下：

| 層級 | 建議表 | 用途 |
|---|---|---|
| 專案 | `projects` | 案名、業主、合約、狀態 |
| 系統 | `systems` | System / Unit / Area |
| 管線 | `lines` | Line No、流體、材質等級、設計條件 |
| 圖面 | `isos` | ISO No、目前版次、圖面狀態 |
| 圖面版次 | `iso_revisions` | Rev、發圖日、作廢/有效、檔案路徑 |
| Spool | `spools` | Spool No、預製狀態、出貨、到場 |
| 焊口 | `welds` | 焊口唯一身分與穩定屬性 |
| 焊口事件 | `weld_events` 或分表 | 完銲、NDE、補焊、PWHT、試壓、請款等歷程 |
| 材料追溯 | `material_traces` | A/B 側母材爐號、銲材批號、MTR |
| 人員/WPS | `welders`, `wps`, `pqr` | 焊工、WPS/PQR 主檔 |
| 商務 | `billing_periods`, `billing_items` | 請款期別、DB 數、狀態 |
| 匯入與稽核 | `import_batches`, `change_log` | 每次匯入來源、差異、人工修改紀錄 |

焊口本體 `welds` 只放「穩定身分」：

```text
project_id
line_id
iso_id
iso_revision_id
spool_id
weld_no
weld_uid
shop_field_flag   -- S/F
joint_type
size
sch
thickness
material_class
diameter_inch
db_qty
created_from_import_batch_id
is_active
```

不要把 NDE 日期、請款狀態、補焊次數、試壓結果全部塞進 `welds`。那些是事件或狀態，應該放在事件表或模組表。

13 張管制表應對應成視圖：

| 管制表 | 實際來源 |
|---|---|
| ISO 圖號管制表 | `isos` + `iso_revisions` + `lines` |
| 焊口管制表 | `welds` + 最新事件彙總 |
| Spool 預製分段 | `spools` + `welds` |
| MTO | `mto_items` + `isos` + `lines` |
| 採購/收料/領料 | `purchase_*` + `mto_items` |
| 爐號/MTR | `material_traces` + `mtr_files` |
| NDE/RT | `nde_requests` + `nde_results` |
| PWHT | `pwht_records` |
| 試壓包/Punch | `test_packages` + `punch_items` |
| 焊工/WPS/PQR | 主檔 |
| 請款/計價 | `billing_items` + `billing_periods` |
| 進度總表 | SQL view / dashboard query |
| MDR | 文件索引 + 完整性檢查視圖 |

需要補的表是 `import_batches`、`change_log`、`iso_revisions`。這三張是避免未來資料互洗、責任不清、版次混亂的保險絲。

## 3. UI 風格:落在光譜哪裡;格子與紀錄的界線

**主張:**  
應落在「試算表式格子 + 紀錄式詳情」中間偏格子，不要做純表單 App，也不要回到純 Excel。

**理由:**  
現場人員習慣橫向掃表、複製貼上、批次修改；但焊口的一生太複雜，不能全部攤在同一張格子裡。

**取捨:**  
格子適合快速比對和批次動作；紀錄頁適合看完整履歷與避免誤改。兩者切開後，使用者會多一個「點進去看詳情」的動作，但換來清楚很多的畫面。

**具體做法:**  

主畫面用「焊口工作表」：

```text
上方: 專案 / 系統 / ISO / Spool / 狀態 快速切換
左側或上方: 進階篩選器
中間: 焊口清單格子
右側: 選取焊口摘要
下方或右側分頁: 基本資料 / 製程 / NDE / 材料 / 請款 / 履歷
```

格子裡只放可掃描、可批次操作的欄位：

```text
ISO No
Rev
Line No
Spool No
Weld No
S/F
Size
SCH
Material
Joint Type
DB
Fit-up
Weld Date
Welder
NDE Method
NDE Result
Repair Count
Test Package
Billing Period
Status
```

不要在格子直接放：

```text
A側爐號完整資料
B側爐號完整資料
多筆銲材批號
完整 NDE 報告歷史
補焊每次原因
PWHT 曲線或報告細節
MDR 文件附件
```

這些放到右側紀錄詳情或事件分頁。

## 4. 可用性

### 4a 輸入(含批次)

**主張:**  
輸入要分成三種模式：單筆快改、批次套用、匯入比對。不要逼使用者每次開 40 欄表單。

**理由:**  
現場常見工作不是「新增一筆完整焊口」，而是「這 80 口今天完銲」、「這批 RT 結果回來」、「第 5 期要請款這些 DB」。

**取捨:**  
批次功能會需要更嚴格的確認畫面，但可以大幅減少手動輸入錯誤。

**具體做法:**  

在清單勾選多筆後提供批次動作：

```text
批次設定完銲日
批次指定打底/蓋面焊工
批次指定 NDE 方法與比例
批次匯入 RT 結果
批次指定試壓包
批次加入請款期別
批次標記已送審 / 已請款 / 已收款
```

批次套用前顯示確認摘要：

```text
即將更新: 126 個焊口
欄位: NDE Result
原本: 93 筆空白, 33 筆 Pending
更新為: Accept
不會覆蓋: 已有 Reject 的 4 筆
```

匯入 Excel 時不要直接覆蓋，改成三步：

```text
1. 上傳 Excel
2. 系統顯示差異: 新增 / 更新 / 衝突 / 將停用
3. 使用者確認套用
```

### 4b 可見性

**主張:**  
2.5 萬筆不能靠「前 300 筆」呈現，要靠狀態分群、彙總數字、例外清單、可展開細節。

**理由:**  
現場管理要先知道哪裡卡住，不是先看完整大表。

**取捨:**  
首頁需要做彙總查詢和快取，但 SQLite 可以處理；必要時為常用欄位加索引即可。

**具體做法:**  

專案首頁先顯示 6 個工作桶：

```text
未組對
已組對未完銲
已完銲待 NDE
NDE Reject / 待補焊
待試壓
可請款 / 已請款
```

每個桶可點進清單，並保留目前篩選條件。

清單不應只顯示前 300 筆，而要：

```text
顯示總筆數: 共 12,483 筆
目前載入: 1-100
支援下一頁 / 無限捲動
支援匯出目前篩選結果
```

另設「例外清單」：

```text
有完銲日但沒有焊工
有 NDE 結果但沒有報告號
已請款但沒有完銲
同一 ISO + Weld No 重複
Rev 已作廢但仍有 active weld
材料爐號缺 A 側或 B 側
```

### 4c 篩選

**主張:**  
篩選器要像查資料，不要只像搜尋框。至少支援欄位條件、日期區間、多選、儲存常用篩選。

**理由:**  
工務、品管、商務會問的問題完全不同。三個篩選條件不可能支撐真實工作。

**取捨:**  
進階篩選 UI 會比現在多一層，但可以保留「簡單搜尋」給日常使用。

**具體做法:**  

篩選區分兩層：

```text
快速篩選:
專案、系統、ISO、狀態、關鍵字

進階篩選:
尺寸
SCH
材質
Joint Type
S/F
Spool
焊工
WPS
NDE 方法
NDE 結果
NDE 日期區間
完銲日期區間
試壓包
請款期別
請款狀態
是否缺資料
是否有補焊
```

條件顯示成可移除的 chips：

```text
System = A1
Size in 2", 3", 4"
NDE Result = Reject
Weld Date = 2026-06-01 ~ 2026-06-30
Billing Period is empty
```

提供「儲存篩選」：

```text
待 RT 結果
本週完銲
第 5 期請款候選
缺爐號
Reject 待處理
```

### 4d 一個焊口太多事件

**主張:**  
一個焊口不要用一張長表單呈現，要用「摘要卡 + 分頁 + 時間線」。

**理由:**  
使用者點開焊口時，最想知道的是：這口是誰、現在到哪、卡在哪、下一步是什麼。不是先看 40 個欄位。

**取捨:**  
事件式呈現比單表單多一點設計成本，但能讓非工程師看懂焊口生命週期。

**具體做法:**  

右側焊口摘要固定顯示：

```text
ISO: 12"-CW-001 Rev.3
Line: CW-12-001
Weld: W-034
Spool: SP-02
S/F: Shop
Size/SCH/Material: 12" / SCH40 / A106
DB: 12
目前狀態: NDE Reject，待補焊
下一步: 建立補焊紀錄
```

下方分頁：

```text
基本資料
製程
NDE / 補焊
材料追溯
PWHT / 試壓
請款
變更履歷
```

時間線範例：

```text
2026-06-02 組對完成
2026-06-03 完銲，打底 A001，蓋面 A008
2026-06-05 RT 委檢，比例 10%
2026-06-07 RT Reject，報告 RT-1142
2026-06-08 建立補焊 R1
```

欄位很多時要分「必填、常用、進階」：

```text
必填: 完銲日、焊工、WPS、狀態
常用: NDE 方法、結果、報告號、請款期別
進階: 爐號、銲材批號、PWHT 曲線、MTR 附件
```

## 5. 最該先決的考量

**主張:**  
最該先解的是「重匯覆蓋整案、洗掉手動輸入」；這會直接破壞使用者信任。

**理由:**  
一旦現場人員發現辛苦補的焊工、NDE、請款資料會被下一次 Excel 匯入洗掉，他們就會回去維護自己的 Excel，系統自然失敗。

**取捨:**  
匯入流程會從「一鍵覆蓋」變成「比對後套用」，速度慢一點，但資料安全高很多。

**具體做法:**  

建立欄位 ownership：

```text
來源欄位: 從 ISO / Excel 匯入，可被新版圖面更新
人工欄位: 現場輸入，不可被匯入覆蓋
系統欄位: 系統計算或狀態推導
```

例如：

```text
可由匯入更新:
ISO No, Rev, Line No, Weld No, Size, SCH, Material, Spool No

不可由匯入直接覆蓋:
Welder, Weld Date, NDE Result, Report No, Billing Period, Material Heat No
```

匯入時做差異表：

| 類型 | 說明 | 預設動作 |
|---|---|---|
| 新焊口 | 新版圖面多出來 | 新增 |
| 既有焊口屬性變更 | Size/SCH/Spool 改變 | 等待確認 |
| Excel 空白但系統有人工資料 | 可能是空白匯入 | 不覆蓋 |
| 舊版有、新版沒有 | 可能刪除或版次變更 | 標記停用，不硬刪 |
| 識別衝突 | 同 key 對到不同資料 | 阻擋匯入 |

識別碼規範建議：

```text
weld_uid = project_id + iso_no + iso_rev_strategy + weld_no + line_no
```

但不要只靠 Weld No，因為不同 ISO 或不同 Line 可能重號。

較穩的唯一鍵：

```text
project_id
iso_no
line_no
weld_no
shop_field_flag
```

若業主圖面版次會重編焊口，則需要人工確認 mapping，不應自動覆蓋。

版次策略：

```text
ISO Rev 是圖面歷史
Weld 是工作物件
新版 ISO 匯入時，系統判斷哪些 weld 延續、哪些新增、哪些停用
舊版資料保留可查，不刪除
```

備份安全最低要求：

```text
每次匯入前自動備份 SQLite
每天第一次啟動自動備份
重大批次修改前自動備份
備份檔名含 project/date/time/action
提供一鍵還原到某次備份
```

## 6. 只能先做 3 件事

**主張:**  
第一階段只做三件事：匯入差異/不覆蓋、進階篩選、焊口右側詳情。這三件最能止血。

**理由:**  
它們分別解決信任、找資料、看懂資料三個核心問題。沒有這三件，後面做再多模組都會變成更大的混亂。

**取捨:**  
暫時不要先做完整採購、完整 MDR、自動排程、大型 dashboard。那些重要，但必須建立在穩定焊口資料之上。

**具體做法:**  

第一件：重做匯入流程。

```text
Excel 上傳
產生差異預覽
標示新增/更新/衝突/停用
人工欄位不覆蓋
匯入前自動備份
匯入後產生批次紀錄
```

第二件：做真正可用的篩選器。

```text
尺寸、材質、型式、焊工、NDE 結果、日期區間、Spool、S/F、期別
支援多選
支援儲存常用篩選
支援匯出目前篩選結果
```

第三件：改掉 40 欄長表單。

```text
清單保留 15-20 個常用欄位
點焊口開右側摘要
詳情用分頁
歷程用時間線
批次修改從清單勾選進入
```

## 7. 我最想強調的一點 + 我可能錯在哪(自我打臉一次)

**主張:**  
我最想強調的是：不要把這個系統做成「比較漂亮的 Excel」，而要做成「資料可靠、看起來像 Excel、操作比 Excel 更不容易錯」的工作台。

**理由:**  
現場人員不是真的愛 Excel，他們愛的是快、看得到、可以複製貼上、錯了能改、主管問時能立刻篩出答案。系統要保留這些優點，同時拿掉多版本、多檔案、互相覆蓋的風險。

**取捨:**  
這代表短期內 UI 看起來可能仍然樸素，不會很像現代 SaaS；但它會更適合少數人、本機、管 2.5 萬焊口的真實場景。

**具體做法:**  
第一版不要追求華麗，追求四個承諾：

```text
匯入不洗掉人工資料
任何焊口都找得到
任何狀態都有原因和下一步
任何大動作都能追溯和還原
```

我可能錯在：如果這家公司其實已經有很強的表單流程與固定責任分工，那麼「偏格子的工作台」可能會讓權限與流程控制不夠嚴謹。那種情況應該更偏「任務表單 + 審核流程」，但仍不建議回到 40 欄大表單。
