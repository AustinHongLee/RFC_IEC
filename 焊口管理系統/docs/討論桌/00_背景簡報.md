# 背景簡報(每位進場的 AI 請先讀)

## 這是什麼

一套「焊口 / ISO 圖面」管理系統,給一家做 **管線預製 + 現場安裝** 的公司(智源)使用。
目的:把原本散落在多個 Excel(焊口管制表、ISO 圖號清單)的資料,整合成單一系統,用網頁 UI 登記、可跨案彙總。

## 技術現況(僅供理解,不是討論重點)

- 後端 Python(FastAPI)+ **單檔 SQLite**;前端 **原生 HTML/CSS/JS(無框架)**;`run.bat` 一鍵啟動,瀏覽器開 `127.0.0.1:8000`。
- 資料層級:**專案 → 系統 → 管線 → 圖面(ISO)→〔spool 預製分段〕→ 焊口**;焊口再連 檢驗(NDE)、請款期別、材料(爐號/銲材)、問題、焊工、WPS。
- 規模感:目前最大一個案子約 **24,879 個焊口、1,717 張圖**;會同時管多個案子(已匯入 5 案)。
- 使用者:現場 / 工務人員,**非工程師**;設計為單機或少數人集中使用。

## 一個「焊口」身上要記多少東西(痛點的根源)

- **規格**:尺寸、厚度、SCH、材質、銲接型式、分類、DB數(吋徑計量)、S(預製)/F(現場)
- **製程追溯**:銲接製程、WPS、打底焊工、蓋面焊工、組對者、組對日期、完銲日期、爐號
- **NDE 檢驗**:方法(RT/UT/PT/MT/VT)、比例、日期、結果、報告號、補焊次數
- **PWHT**:是否需要、是否完成、日期
- **試壓**:試壓包、日期、結果
- **狀態與商務**:狀態、承包商、請款期別、請款狀態
- **材料**:A 側母材爐號、B 側母材爐號、銲材批號(可多筆)

→ **一個焊口要登記 / 追蹤的事件非常多。這就是「難受」的來源。**

## 目前 UI 長怎樣(被嫌的地方)

- 「焊口登錄」是一張長表格;點任一列會跳出一個**很長的 Modal 表單**,一次把上面 ~40 個欄位(分成幾個小標題:基本資料 / 製程與追溯 / 檢驗 / PWHT 試壓 / 狀態商務)全攤在眼前,下面再接「檢驗紀錄」「材料」兩個子表。**一次面對全部,壓力很大。**
- **篩選只有三個**:一個搜尋框(銲口編號 / 圖號 / 材質)+ 狀態下拉 + 系統下拉。
  想用「尺寸、材質、焊接型式、焊工、NDE 結果、日期區間、spool、S/F、請款期別…」來精確找,**做不到**。
- 清單一次只顯示前 300 筆。**2.5 萬筆的可見性與查找明顯不夠。**

## 使用者要的(北極星)

**好輸入、直觀、看得見、完整且精確的篩選(想找什麼就篩得到)、不要被「一個焊口那麼多事件」淹沒。**

## 邊界(請尊重)

- 使用者**非工程師** → 方案要能落到「現場真的會用」。
- 維持**輕量**(單檔 SQLite + 原生前端)→ 除非有強理由,別要求換 React/大框架/雲端。
- 重點是**可用性與資訊設計**,不是再堆功能。
