## 一句話結論

**這套系統的當務之急不是再加功能,而是「讓人敢信任 + 找得到 + 看得懂」。**
三家一致把「匯入會洗掉手動資料」列為頭號要解;接著是「真正的篩選」與「把 40 欄大表單拆掉」。
所以——**剩下的企業級功能(採購、MDR)往後排,先把可用性與資料信任做起來。**

---

## A. 三家共識(視為已拍板,不再爭)

1. **資料模型 = 一份資料、多種視圖。**
   13 張管制表大多不是 13 張獨立表,而是用 SQL VIEW 從骨幹(專案→系統→管線→圖面→spool→焊口 + 事件子表)投影出來。同一個焊口只存一份,各表只是不同看法。`v_joint_full` 已是雛形。

2. **NDE 摘要欄是「快取」,不是真相。**
   `weld_joint.nde_*` 與 `inspection` 子表同一件事存兩份(補焊 2 次就有 3 筆 RT)。真相在 `inspection`,焊口上的摘要欄只作清單/篩選用的快取。

3. **UI = 清單用「格子」,單一焊口用「紀錄/分區」。**
   清單像試算表(≤12–15 欄、顏色狀態、可排序);點一筆**不要跳 40 欄大 Modal**,改成右側抽屜 / 分頁 / 摺疊。三家都要求拆掉現在的 `jointFields()` 一次全開。

4. **輸入分三條路:匯入建骨架 → 批次補肉 → 單筆精修。**
   現場不是「一筆筆新增完整焊口」,而是「這 80 口今天完銲」「這批 RT 回來了」。要能**勾選多筆一次填**(完銲日/焊工/期別/狀態)。

5. **匯入改「合併」,永不整案覆蓋。**(最關鍵,見 C 區)

6. **篩選擴到 12–15 維、可組合、可存常用視圖。**
   下拉帶數量(A106 (1,847))、多選、條件 chips、URL 可分享。後端 `list_joints` 已支援一半參數。

7. **去掉「只顯示前 300 筆」。**
   改分頁 / 載入更多 / 虛擬捲動,並顯示總筆數與「篩選結果統計列」。

8. **識別碼要規範化 + 業務唯一鍵。**
   全形→半形、連接符統一、流水號轉整數;焊口唯一 = `(project_id, drawing_id, joint_no)`。

9. **版次別過度設計。** `drawing_revision` + `audit_log` 已足夠;焊口不做版本表,改了留 log。

10. **備份要有紀律。** 匯入前自動備份、DB 不進 git(或關 app 再備份)、整案匯出 Excel 當人類可讀備援。

## B. 分歧 → 我的裁決

| 爭點 | 三家立場 | ❖ 我的裁決 | 理由 |
|---|---|---|---|
| 試壓包 | DeepSeek/ChatGPT:獨立表;Grok:欄位+視圖 | **獨立表 `test_package`**,但排到後面(非前三) | 試壓包是跨焊口的管理單元,要能反查「這包有哪些焊口」,字串欄做不到。但不是當前阻斷點。 |
| NDE 摘要 | DeepSeek:用 VIEW 推導;Grok:trigger 同步快取 | **保留快取欄,inspection 為唯一編輯入口,寫入時同步**(現行 `add_inspection` 已這樣做) | 2.5 萬列要靠索引篩 NDE 結果;每列即時 MAX 子查詢會慢。快取欄 + 子表為真相,兩全。 |
| 格子內逐格編輯 | Grok 自我打臉:原生 table 的貼上/Undo/填滿很難、對非工程師反而難教 | **以「選多筆 → 批次面板」為主,逐格編輯只留 1–2 個高頻欄**(狀態用既有「推進▶」) | 無框架下硬做 Excel 式編輯不穩、難教、難維護;批次面板更簡單、更可靠。 |
| 時間軸 vs 摺疊表單 | DeepSeek 自我打臉:事件稀疏時時間軸沒料 | **先做「分區摺疊抽屜」(Phase A),時間軸當 Phase B** | 剛匯入的焊口只有一條「規劃」,時間軸沒價值;摺疊表單立刻可用。 |
| 虛擬捲動 vs 分頁 | 兩者皆可,虛擬捲動較理想但較費工 | **先做「分頁 / 載入更多」,虛擬捲動列為後續優化** | 先用最穩、最少程式的做法解掉 300 上限;真的卡再上虛擬捲動。 |

## C. 最終優先順序(裁決)— 重新排過

> 把原本的「繼續加企業功能(採購/MDR)」**降級**,先做下面這三件「止血」。三家一致。

### P0 — 匯入改「合併模式」,不洗掉手動資料 ★最優先★
- **問題:** `importer.py` 現在 `replace=True` → `DELETE FROM project` 整案砍掉重來,手填的焊工/檢驗/材料/請款全滅。這是「為什麼大家寧可回去用 Excel」的根因(信任問題)。
- **做法:** 改 UPSERT,鍵 `(project_id, drawing_id, joint_no)`;**手動欄位白名單永不被覆蓋**(焊工、fitup、pwht_*、test_*、billing、claim、remark、nde_*);匯入後出**差異報告**(新增 N / 更新規格 M / 保護手動 K / 警告)。整案重來改成獨立的「危險:重建專案」按鈕。
- **底子已有:** `merge_completion_file()` 就是增量合併的雛形,把它升格成預設路徑。

### P1 — 焊口清單:完整篩選 + 去掉 300 上限 + 內建視圖
- 篩選擴到 10+ 維(尺寸/材質/型式/S·F/焊工/WPS/NDE結果/日期區間/spool/期別…),多選、帶數量、chips、可存「我的待辦/本週完銲/RT不合格/未請款」。
- 後端補索引(size, material, shop_field, weld_date, welder_*),`list_joints` 擴參數(全部參數化)。
- 移除前端硬編 `limit=300`,改分頁/載入更多 + 顯示總筆數與統計列。

### P2 — 焊口詳情改「抽屜 + 分區摺疊」+ 清單精簡 12 欄 + 批次面板
- 點一筆 → 右側抽屜,頂部固定身分證+狀態,下面分區(規格/製程/NDE/材料/試壓/商務/歷程)預設摺疊。
- 清單精簡到 ~12 欄;勾選多筆 → 批次填(完銲日/焊工/期別/狀態)。
- 保留「推進▶」。

### P3 以後(先擱)
試壓包獨立表、時間軸化、虛擬捲動、採購鏈、MDR 組卷。重要,但都建立在 P0–P2 穩了之後。

## D. 對到現有程式碼的落點(給實作)

- **P0:** `importer.py::import_weld_control` 的 `replace`/`DELETE` → UPSERT + 白名單;沿用 `merge_completion_file` 的比對邏輯;`weld_joint` 加 `UNIQUE(project_id, drawing_id, joint_no)`(需處理 drawing_id NULL)。
- **P1:** `app.py::list_joints` 擴查詢參數 + `schema.sql` 補索引;`static/app.js::renderJoints` 篩選列與分頁;`lookups` 回傳各維度選項+COUNT。
- **P2:** `static/app.js::jointFields()/editJoint()` 由大 Modal 改抽屜+摺疊;清單欄位精簡;新增批次面板與 `POST /joints/batch`。

## E. 各家最值得記的獨到點(存證)

- **ChatGPT:** 「欄位三分類(來源/人工/系統)」、匯入差異表、**例外清單**(有完銲日無焊工、已請款無完銲、重複焊口…)——很好的資料健檢。
- **DeepSeek:** 直接引用真實行號點出「雙寫原罪」與 `replace=True` 位置;主張 `pwht_event` 獨立表;**Phase A/B 漸進**(先摺疊、後時間軸)的務實修正。
- **Grok:** 一句定位「**系統是焊口管制表的線上版,Excel 是 IO 埠**」;`PRAGMA journal_mode=WAL` + 備份前 CHECKPOINT;誠實打臉「格子逐格編輯對非工程師不一定友善」。
